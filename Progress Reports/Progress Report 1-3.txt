# WormCraft / Soil to Sun — Progress Summary

## Thread Summary: VS003 Completion, VS004 Visual/Interaction Pass, and DeepSeek Workflow Discovery

## High-Level Result

This thread moved the project from a crude but functional worm/dirt prototype into the beginning of a readable game slice.

The biggest accomplishments were:

* Dirt placement became functional.
* The world moved from 16px tiles toward 32px terrain tiles.
* A new animated brain-worm sprite was integrated.
* Dirt rendering now uses actual texture art instead of only debug colors.
* Target preview was implemented.
* Dig damage feedback was implemented.
* Natural dirt durability was raised to 24 hits to make manual digging feel intentionally slow.
* DeepSeek/Ziva workflow was proven viable for cheap, surgical Godot edits when tightly constrained.

The project now has the first real signs of a playable loop: the worm can move, dig, place dirt, see what it is targeting, damage dirt over time, and interact with a more visually readable world.

---

# 1. VS003 — Dirt Placement and Tunnel Shaping

## Dirt Placement Added

A `place_dirt` action was added and bound to **F**.

The player can now spend Dirt Pile from inventory to place dirt into the tile in front of the worm.

Placed dirt is treated differently from natural dirt:

* Natural dirt: 24 digs, +5 Dirt Pile when cleared
* Placed dirt: 4 digs, +1 Dirt Pile when cleared

This prevents dirt duplication loops.

## Placement Debugging

Initial reports claimed placement was working, but playtesting showed it was not. The issue was that `place_dirt` was not actually present in the Input Map.

This confirmed that human playtesting is necessary. Ziva/agent reports cannot be trusted without manual verification.

## Inventory / Status Flyout Fixes

An issue was found where the Inventory flyout opened with **I** but did not close. This violated the minimal/flyout UI philosophy.

The fix updated the flyout logic so toggling immediately updates panel position after changing `is_open`.

Status flyout behavior was also corrected similarly.

---

# 2. Shift from Ziva/Haiku to DeepSeek Micro-Edits

## Cost Discovery

Using Ziva with stronger models cost more than expected for small tasks. Some simple script replacements or minor actions cost several dollars.

DeepSeek through Ziva was later tested and proved dramatically cheaper, sometimes costing less than a cent for useful changes.

## Workflow Change

The project workflow shifted to:

* ChatGPT: architecture, diagnosis, prompt design, code review
* DeepSeek/Ziva: cheap Godot implementation intern
* User: playtester, visual judge, commit gatekeeper

This workflow proved effective when prompts were extremely narrow.

## Key Lesson

DeepSeek works if:

* one task is given at a time
* allowed files are specified
* forbidden files are specified
* the exact behavior is described
* unrelated refactors are explicitly forbidden
* the user playtests afterward

DeepSeek should not be asked to design, improvise, or broadly “fix” a system.

# 4. Tile Art / Dirt Rendering

## 16px Tiles Reconsidered

The project originally used 16px tiles for Terraria-like density, but generated art from Ludo looked much better at higher resolution.

We decided the game does not need to look like an NES game. The tile scale was moved toward **32px**.

## Dirt Texture Pipeline

Ludo produced large source images, often 768x768 or 1024x1024. These cannot be used directly as game tiles.

The correct pipeline became:

* Large image = source art
* 32x32 PNG = actual game asset

A 32x32 dirt tile was generated/resized and placed in:

`res://assets/tiles/dirt.png`

## Atlas Failure

DeepSeek/Ziva initially tried to treat the large dirt image as an atlas. This produced broken visuals: random chunks, gray blocks, and black gaps.

Diagnosis: the renderer sampled arbitrary regions from a large texture instead of drawing a single 32x32 tile.

## TileMapLayer Split-Brain Problem

A later attempt introduced `TileMapLayer` rendering while gameplay still used `tile_grid`.

This caused a split-brain world:

* visuals came from TileMapLayer
* gameplay came from `tile_grid`

Symptoms included:

* worm walking through visible walls
* digging saying “Nothing to dig”
* visible dirt not disappearing when dug

Fix:

* Removed TileMapLayer path
* Removed TileSet/atlas dependency
* Restored `tile_grid` as the single source of truth
* Manual `_draw()` rendering now draws from `tile_grid`
* Dirt texture is drawn for `TileType.DIRT`
* Other tile types fall back to debug colors unless textured

This was one of the most important architectural corrections in the thread.

## Multiple Dirt Variants

Dirt variants were introduced conceptually:

* `dirt.png`
* `dirt2.png`
* `dirt3.png`

All are still `TileType.DIRT` behaviorally.

They should be selected randomly or deterministically per cell for visual variety, without creating new tile types.

Key rule: variant selection must be stable per tile and must not randomize every frame.

## Transparent Edge Issue

Some Ludo textures produced transparent edges/padding. Godot showed gray behind those transparent pixels.

Resolution:

* Full terrain tiles should be opaque.
* Transparency is appropriate for sprites, cracks, particles, and UI, not for full dirt/rock/surface blocks.
* Terrain assets should be 32x32 opaque PNGs with no transparent margins.

# 5. Worm Sprite / Animation Integration

## Brain-Worm Character Direction

A protagonist visual direction was developed:

* sapient earthworm
* comically enlarged visible brain
* brain integrated into flesh
* weird but charming
* not humanoid
* no arms/legs/weapons
* head/front must be readable

Several Ludo outputs were evaluated. The final direction used a more worm-like body with a visible enlarged brain.

## Sprite Sheet

A 6x6 sprite sheet of the brain-worm was generated. It was visually promising but not directly game-ready because:

* too many frames
* inconsistent spacing
* not cleanly packed
* oversized compared to original 48x16 target

Still, it worked as source material.

## Animated Sprite Integration

The animated worm sprite was integrated.

Sprite-facing behavior uses separate left/right textures rather than `flip_h`.

The worm visually looked much better, but introduced alignment issues due to sprite size and origin mismatch.

## Wall / Offset Issue

The worm appeared to overlap walls because its sprite body extended beyond the logical collision/tile point.

A directional offset fix was discussed: if using flipped sprites, X offset must mirror. However, the current system uses separate left/right textures.

## Ground Floating Issue

The worm appeared to float above the floor.

Initial DeepSeek fix tried to align sprite bottom with collision math. That did not solve the visual issue.

We corrected the prompt to emphasize viewport/visual confirmation instead of math.

Final successful visual offset:

`Sprite2D.offset = Vector2(-40, -2.5)`

This made the worm appear visually grounded in the pocket.

Key lesson:

**For visual alignment, mathematical collision alignment is not enough. Viewport-based human feedback is required.**

# 6. Spawn / Pocket Issues

After moving to 32px tiles, the worm appeared to start near the surface.

Diagnosis: likely old pixel-based spawn position did not match new tile size.

Fix direction:

* `LayeredTestWorld` exposes start position via grid coordinates.
* The worm should spawn using `world.grid_to_world_center(Vector2i(48, 20))`.
* This prevents future tile-size changes from breaking spawn position.

The worm now starts in the underground pocket.

The pocket is larger because 32px tiles make each tile physically larger. That is acceptable for now.

## Toggle Added

* Q turns target preview off

# 8. Dig Durability Tuning

## Natural Dirt Changed to 24 Hits

Base worm digging was changed:

* Natural dirt: 24 hits
* Surface: 24 hits
* Placed dirt: 4 hits

Design reason:

Manual digging should feel slow and tedious enough that the player wants to invent tools, build machines, or explore for better materials.

The base worm should not be an efficient mining machine.

This change was made before adding damage visuals so the damage overlay could be designed around the correct values.

# 9. Procedural Dig Damage Overlay

## Implemented

A procedural damage overlay was added.

When a tile is partially dug, cracks/marks appear over it.

The overlay:

* uses `tile_durability`
* draws procedurally with lines
* intensifies as the tile nears breaking
* draws over dirt/surface/placed dirt
* disappears when the tile becomes empty
* renders below the target preview

The first version is visually crude but functional.

It confirms the data/rendering pipeline works.

## Current Behavior

* Natural dirt confirms `Digging dirt: 1 / 24`
* Damage appears before clearing
* Target preview still draws on top
* No gameplay logic was changed

This passes the prototype requirement even though the art can be improved later.

# 1. Backstory / Game Identity Clarified

## Core Premise Refinement

The worm has been alive since the 1980s.

He has lived in the backyard of a tax attorney for decades.

He is the first and only human-level intelligent earthworm in the universe, and he knows it.

He is bored, lonely, paranoid, arrogant, and fragile.

The backyard is not a cute setting. It is a dangerous world of giants, brutes, grass-pillars, falling objects, and scavenged relics.

The worm has already built a few machines before the game starts.

The game begins in medias res, with the worm already living in a small workshop pocket.

## Long-Term Goal

The worm’s ultimate goal is to escape Earth and begin a worm civilization on a low-gravity body such as an asteroid, moonlet, or planetoid.

Eventually, he will need to find a suitable mate and use advanced biological manipulation to create intelligent offspring.

This long-term goal is not for early access.

Early access should focus on:

* backyard survival
* burrow expansion
* storage
* scavenging
* first machines
* first brute interactions
* surface risk
* early ecosystem manipulation

## “Brutes”

Ordinary arthropods are called “brutes” by the worm.

This term is good and should remain.

It expresses the worm’s superior intelligence and contempt while also highlighting his physical vulnerability.

Examples:

* ants: disciplined brutes
* beetles: armored brutes
* spiders: silk brutes
* crows: sky brutes with negotiable intelligence
* humans: distant, blurry, incomprehensible giants

# 2. Visual Scale Rule

A major visual rule was established:

Everything must be shown from worm scale.

This means:

* a blade of grass is almost as thick as the worm
* pebbles are boulders
* a child’s bead is a major relic
* plastic scraps are powerful synthetic materials
* spider silk is industrial cable
* humans should be distant, blurry, partial, or implied
* boots, mower blades, watering cans, and shadows matter more than human faces

Humans should not appear as normal readable characters unless the game intentionally breaks scale for horror.

This rule should influence all future art prompts and level design.

# 4. Quantum Space Folder

## Design Purpose

The Quantum Space Folder is the worm’s first functional advanced invention.

It explains why the worm can store and place soil at all.

But it is limited, because the worm is materially poor and cannot simply solve storage from the start.

This creates pressure for future wall satchels, storage hooks, chests, and infrastructure.

## Implemented Behavior

Dirt Pile capacity was added.

Reported value:

* `DIRT_PILE_CAPACITY = 30`

Inventory now displays:

`Dirt Pile: X / 30`

Both InventoryFlyout and StatusFlyout were updated.

## Capacity Behavior

The intended behavior:

* natural dirt can add Dirt Pile
* placed dirt can add Dirt Pile
* Dirt Pile cannot exceed 30
* if a tile is cleared while full, the tile still clears
* excess soil is wasted
* digging is not blocked by capacity
* placement still spends 1 Dirt Pile
* eating still spends 1 Dirt Pile and restores hunger

The user playtest confirmed the important mechanical behavior:

* Dirt Pile did not exceed `30 / 30`
* placed dirt cleared correctly
* placed dirt did not create a duplication loop

## Minor Known Issue

The console did not clearly print the expected “soil wasted” message when clearing placed dirt while the folder was full.

This is a feedback bug, not a mechanical failure.

It should be added to a later micro-cleanup:

Capacity Feedback Cleanup:

* if resource cannot fit, print `Folder full. Soil wasted.`
* if some fits, print `Folder full. Added X, wasted Y soil.`
* ensure `last_action` matches the printed message

This should not block VS005 or VS006.

# 5. VS006 Planning — Fall Damage Split Into Parts

## Initial Design

The worm is extremely fragile in its natural state.

A fall of three tile-spaces should kill him.

This makes vertical space dangerous and gives future value to:

* armor
* ramps
* bridges
* placed dirt
* wall satchels
* safe tunnel planning
* movement tools
* machines

## Why VS006 Was Split

Fall damage is dangerous to implement because the movement/physics system is still prototype-level.

If lethal damage was enabled immediately, bad fall counting could create unfair deaths.

Therefore VS006 was split into staged parts:

1. Part 1: fall tracking only, no death
2. Part 2: spawn safety and fall test geometry
3. Parts 3+4: lethal fall damage and death screen

This was the right decision.

# 6. VS006 Part 1 — Fall Tracking Instrumentation

## Purpose

Part 1 added fall tracking without killing the worm.

The goal was to measure fall distance first.

## Files Edited

Reported files:

* `res://scripts/player/worm_player.gd`
* `res://scripts/ui/status_flyout.gd`

## Added Constants

Added to `worm_player.gd`:

* `FALL_TRACKING_ENABLED`
* `LETHAL_FALL_TILES = 3`
* `WARNING_FALL_TILES = 2`

At this stage, lethal fall distance was detected but not applied.

## Added State

Added:

* `is_falling`
* `fall_start_grid_y`
* `last_fall_distance_tiles`

## Fall Detection

A fall begins when the tile below the worm is passable and the worm was not already falling.

A landing occurs when the tile below becomes solid again while `is_falling` is true.

Fall distance is calculated by grid Y difference:

`current_grid_y - fall_start_grid_y`

## StatusFlyout Update

StatusFlyout was updated to show:

`Last Fall: X tiles`

## Part 1 Discovery

Part 1 revealed a serious issue:

The worm spawned into a 3-tile fall.

Console showed:

`Fell 3 tiles. Lethal distance detected.`

This meant lethal fall damage could not be enabled yet.

If death had been enabled at this stage, the worm would have died on game start.

Part 1 succeeded because it caught the problem before it became fatal.

# 9. Commit / Git Issue

## Problem

While committing VS006, `git add .` failed with:

`error: invalid path 'nul'`

Git showed an untracked file:

`nul`

On Windows, `nul` is a reserved device name.

Git refused to add it.

This was probably created accidentally by DeepSeek/Ziva or by a bad shell redirect.

## Attempted Fix

PowerShell command attempted:

`Remove-Item -LiteralPath "\\?\C:\GodotProjects\worm-craft\nul" -Force`

But `git status` still showed `nul`.

## Workaround

The successful approach was:

1. Add real files explicitly instead of using `git add .`
2. Commit the intended files.
3. Add a local exclude rule for `nul`.

The local exclude apparently worked.

## Lesson

Avoid relying on `git add .` when junk files appear.

Use explicit `git add` commands when needed.

If `nul` appears again, handle it locally or exclude it.
